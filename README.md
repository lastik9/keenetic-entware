# keenetic-entware

**Русский** · [English](README.en.md)

Подготовка USB-флешки под **Entware** для роутеров **Keenetic / Netcraze** — одной командой, на «голом» маке, без Homebrew, Docker и сторонних программ разметки.

Скрипт размечает флешку (swap + ext4), форматирует ext4, кладёт на неё установщик нужной архитектуры и сам проверяет результат. Дальше остаётся вставить флешку в роутер и включить OPKG.

## Зачем

macOS не умеет создавать ext4 своими средствами. Обычные инструкции требуют ставить Paragon/сторонние программы или запускать Docker. Здесь всё делается одной командой: скрипт скачивает самодостаточные `mke2fs`/`debugfs` (universal, зависят только от `libSystem`), так что **на маке ничего ставить не нужно**.

## Требования

- macOS (Apple Silicon или Intel)
- USB-флешка (**всё содержимое будет стёрто**)
- Интернет на маке (скачать установщик) и на роутере (установить пакеты Entware)

Homebrew **не нужен**. Если `e2fsprogs` из Homebrew уже стоит — скрипт возьмёт его; если нет — скачает готовые universal-бинарники из Releases.

## Использование

Запусти в Терминале:

```
bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/prepare.sh)
```

Скрипт:
1. Покажет **только съёмные** накопители и попросит выбрать нужный (с подтверждением вводом имени — чтобы нельзя было стереть не тот диск).
2. Спросит архитектуру процессора роутера (с подсказкой по моделям).
3. Разметит, отформатирует ext4, запишет установщик и проверит результат.

### Холостой прогон (ничего не пишется)

Посмотреть, что именно будет сделано, не трогая диск:

```
DRY_RUN=1 bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/prepare.sh)
```

### Принудительно путь «голого мака» (игнорировать Homebrew)

```
IGNORE_BREW=1 bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/prepare.sh)
```

## На роутере

После того как флешка готова:
1. Вставь флешку в Keenetic.
2. Веб-интерфейс → **Общие настройки** → включи компоненты **«Поддержка открытых пакетов (OPKG)»** и **«Файловая система Ext»**.
3. Открой страницу **OPKG** → выбери накопитель **OPKG** → **Сохранить**. Роутер распакует установщик и докачает пакеты Entware с `bin.entware.net` (роутеру нужен интернет).
4. Смотри **Системный журнал** (Диагностика) — там появятся строки об успешной установке Entware.

> **Важно: перезагружай роутер при смене флешки.** Если вынул одну флешку и вставил другую, обязательно перезагрузи роутер перед установкой. Иначе роутер держит следы прошлой флешки (буквы дисков, старую конфигурацию OPKG), и установка может упасть с ошибкой `exec format error` в журнале. Правило простое: **одна флешка + перезагрузка = чистый старт.**

> **Если видишь `exec format error`** (Entware «встал», но не запускается / SSH не пускает): перезагрузи роутер с одной вставленной флешкой — почти всегда это лечит. Если ошибка повторяется стабильно на конкретной флешке — сверь архитектуру: на рабочей установке команда `opkg print-architecture` покажет реальную арку (например, `mipsel-3.4`). Не полагайся на `show version` из веб-CLI — он показывает арку в широком смысле (`mips` для mipsel-систем) и вводит в заблуждение. Учти также, что **Netcraze (NC-XXXX)** — это отдельный бренд Keenetic: у одинаково названных моделей (например, Viva) арка может отличаться, поэтому `opkg print-architecture` — самый надёжный источник.

## Два SSH: порт 222 и порт 22

На роутере с Entware живут **два независимых SSH-сервера**, и это важно понимать:

| | порт **222** | порт **22** |
|---|---|---|
| Кто это | `dropbear` из Entware (`/opt/sbin/dropbear`) | штатный SSH Keenetic (`/usr/sbin/dropbear`) |
| Где живёт | на флешке (`/opt`) | в прошивке роутера |
| Логин | `root`, пароль по умолчанию `keenetic` | **пользователь из веб-панели** (обычно `admin`) |
| Куда попадаешь | сразу в шелл BusyBox | в CLI Keenetic — далее команда `exec sh` |

**Порт 22 — аварийный вход.** Не отключай встроенный SSH-сервер Keenetic: если Entware-овый dropbear на 222 не поднялся (флешка не смонтировалась, пакет обновился, протух pid-файл), это единственный способ попасть на роутер по SSH:

```
ssh -p 22 admin@192.168.10.1      # логин/пароль как в веб-панели
exec sh                            # из CLI Keenetic проваливаемся в BusyBox
```

Обрати внимание: под `root` штатный SSH на 22 **не пустит** ни с каким паролем — он не знает такого пользователя. `root`/`keenetic` — это учётка только для Entware-овского dropbear на 222.

### `Connection refused` на порту 222 после перезагрузки

Симптом: после ребута `ssh -p 222` даёт `Connection refused`, но если зайти через порт 22 и выполнить `/opt/etc/init.d/S51dropbear start` — SSH тут же работает. И так каждый раз.

Причина — **баг в init-скрипте пакета dropbear** из Entware. Штатный `S51dropbear` считает демон живым по формуле:

```sh
[ -f $PIDFILE ] && [ -d /proc/`cat $PIDFILE` ]
```

то есть проверяет только **существование** каталога `/proc/<PID>`, не сверяя, **чей** это процесс. Сам pid-файл (`/opt/var/run/dropbear.pid`) лежит на ext4 флешки и **переживает перезагрузку**. После ребута его номер занимает другой процесс (на Netcraze это `tsmb-server` — встроенный SMB-сервер), скрипт решает «уже запущен», молча выходит — и на 222 никто не слушает. Ребут не помогает: pid-файл каждый раз тот же.

Проверить у себя (через порт 22 → `exec sh`):

```sh
cat /opt/var/run/dropbear.pid                       # например 634
ps w | grep '[/]opt/sbin/dropbear'                  # пусто — демона нет
cat /proc/$(cat /opt/var/run/dropbear.pid)/comm     # tsmb-server — вот и виновник
```

**Лечение** — пункт **4** в меню `router-setup.sh` («Только фикс SSH-222»). Он заменяет `dropbear_status` на проверку `/proc/<PID>/comm` (процесс действительно `dropbear`) и удаляет протухший pid-файл перед стартом. Оригинал сохраняется в `S51dropbear.bak`. Патч также накатывается автоматически в режиме **1** (полная настройка).

> **После `opkg upgrade dropbear` патч затирается** — пакет ставит свой init-скрипт. Прогони пункт 4 заново.

Готовый файл лежит в репозитории: [`S51dropbear`](S51dropbear).

### Настройка роутера (swap + подготовка под XKeen)

После установки Entware зайди на роутер по SSH (логин `root`, пароль `keenetic`, порт `222`) и запусти хелпер. На чистом Entware ещё нет HTTPS-клиента, поэтому сначала ставим `wget-ssl`, потом качаем скрипт — **две команды**:

```
opkg update && opkg install wget-ssl ca-bundle ca-certificates
wget -qO- https://raw.githubusercontent.com/lastik9/keenetic-entware/main/router-setup.sh | sh
```

Хелпер покажет **меню** из четырёх сценариев:

1. **Полная настройка** — активирует **swap** (`mkswap` + автозапуск `S02swap`, поиск раздела по метке `SWAP`, устойчиво к смене буквы диска), обновит opkg и поставит базовые пакеты (`curl`, `tar`, `nano`, `vim`, `ca-bundle`), проверит компоненты роутера (netfilter, IPv6), накатит фикс `S51dropbear`, предложит сменить SSH-пароль и **сразу запустит установщик [XKeen](https://github.com/jameszeroX/XKeen)** (GitHub, при недоступности — зеркало jsDelivr).
2. **Только установка XKeen** — если Entware и swap уже настроены, нужен только прокси. Тот же fallback GitHub → jsDelivr.
3. **Только swap** — быстрый путь после восстановления флешки из бэкапа: проверяет и включает swap (с человекочитаемой диагностикой и размером), настраивает автозапуск — ничего лишнего не переустанавливая.
4. **Только фикс SSH-222** — чинит протухший pid-файл dropbear (см. раздел выше). Пригодится после `opkg upgrade dropbear`.

После настройки — перезагрузи роутер (`reboot`). Swap поднимется автоматически; проверить можно командой `cat /proc/swaps`.

### Geodata для XKeen/Mihomo (если базы не качаются)

При старте XKeen/Mihomo скачивает базы `geoip`/`geosite`/`mmdb` с GitHub (release-assets). Из некоторых сетей (например, из РФ) GitHub может быть недоступен — в логах виден таймаут TLS, и прокси не поднимается без баз:

```
ERRO can't initial ASN: can't download ASN.mmdb: net/http: TLS handshake timeout
FATA Parse config error: rules[121] [IP-ASN,16509,PROXY] error: ...
```

Решение — прописать зеркала через CDN **jsDelivr** в конфиге Mihomo (`/opt/etc/mihomo/config.yaml`):

```yaml
geox-url:
  geoip: "https://fastly.jsdelivr.net/gh/Loyalsoldier/v2ray-rules-dat@release/geoip.dat"
  geosite: "https://fastly.jsdelivr.net/gh/Loyalsoldier/v2ray-rules-dat@release/geosite.dat"
  mmdb: "https://fastly.jsdelivr.net/gh/P3TERX/GeoLite.mmdb@download/GeoLite2-Country.mmdb"
  asn: "https://fastly.jsdelivr.net/gh/P3TERX/GeoLite.mmdb@download/GeoLite2-ASN.mmdb"
```

После правки перезапусти XKeen/Mihomo — базы подтянутся с зеркала.

### «XKeen долго стартует» — это не флешка

Строка `Initial configuration complete, total time: 47179ms` (а иногда и 100+ секунд) пугает, но **износ флешки, swap и скрипты этого проекта ни при чём**. Mihomo при старте разбирает geodata в память, и самый жирный кусок — правило `category-ads-all => REJECT` (больше 160 000 записей). Это **чистая нагрузка на CPU**, а роутеры Keenetic/Netcraze на MT7621A имеют всего 880 МГц. Разные прогоны на одном и том же диске дают разное время — значит дело не в вводе-выводе.

Замеры на живом железе (Netcraze Viva, 64 ГБ USB 2.0, XKeen работает):

```
dd if=/dev/sda2 of=/dev/null bs=1M count=200   →  ~37 МБ/с (потолок USB 2.0)
free                                            →  swap used = 0, свободно ~120 МБ RAM
```

Флешка читает на пределе шины, swap не задействован вообще. Хочешь ускорить старт — убирай тяжёлые правила из конфига Mihomo, а не меняй накопитель.

## Правка конфигов по сети (SMB), без SSH

Не обязательно лезть в терминал, чтобы поправить `config.yaml` у Mihomo. Раздел `/opt` с флешки можно **открыть как обычную сетевую папку** — в Finder или Проводнике — и редактировать файлы двойным кликом.

Ставить Samba из Entware для этого **не нужно**: в Keenetic/Netcraze уже встроен SMB-сервер (`tsmb-server`). Нужно лишь корректно настроить общий ресурс.

### Настройка (веб-панель)

Открой **Приложения → Сервер SMB**:

1. **Удали мёртвые общие ресурсы.** Если под записями красным написано «Папка не найдена» — это следы прошлых флешек, они мешают создать рабочую шару. В журнале роутера они выдают себя строками:
   ```
   [E] Cifs::ServerTsmb: share record "OPKG" already exists.
   [W] Cifs::ServerTsmb: failed to automount "<UUID>", ignored.
   ```
   Удали все такие записи корзинкой. Шара для раздела `SWAP` тоже не нужна — это не файловая система.

   > Откуда берутся: при каждой подготовке флешки `diskutil` сперва создаёт разделы как FAT32, и роутер запоминает их короткие серийники (`45D2-1B0A`). Меняешь флешку — старая запись остаётся. Посмотреть весь список можно в CLI (`ssh -p 22 admin@<ip>`) командой `show running-config`, секция `cifs`.

2. **Сними галку «Анонимный доступ».** В режиме анонимного доступа (в конфиге это `cifs permissive`) сервер не проверяет учётные записи, и вход под `admin` отбивается с `Authentication error`. Плюс это дыра: `/opt` — корень Entware с конфигами, init-скриптами и SSH-ключами — оказывается доступен любому в локальной сети без пароля.

3. **Добавь общий ресурс** кнопкой «+ Добавить общий ресурс», указав текущий подключённый раздел `OPKG`. Проверить, что он смонтирован, можно в **Приложения → Диски и принтеры** (статус «Подключено», ФС `EXT4`).

4. Убедись, что у пользователя есть право **«Сетевые ресурсы SMB»**: **Управление → Пользователи → admin**.

### Подключение

**Finder (macOS):** `Cmd+K` → `smb://192.168.10.1` → логин и пароль от веб-панели.
**Проводник (Windows):** в адресной строке `\\192.168.10.1`.

Конфиг Mihomo окажется по пути `etc/mihomo/config.yaml` внутри шары.

Из терминала то же самое:

```
smbutil view //admin@192.168.10.1                       # список шар: должна быть OPKG
mkdir -p ~/smb-opkg
mount_smbfs //admin@192.168.10.1/OPKG ~/smb-opkg        # смонтировать
ls ~/smb-opkg/etc/mihomo/
umount ~/smb-opkg                                       # отключить
```

> **Гостевой вход не поможет.** Современные macOS отказываются от анонимных SMB-подключений, а сервер в режиме `permissive` показывает гостю только служебный `IPC$` без единой файловой шары. Работает именно связка «анонимный доступ выключен + вход под `admin`».

### Осторожно при правке

- Перед изменением сделай копию рядом: `config.yaml.bak`. Откатиться проще, чем искать ошибку отступа в YAML на роутере.
- Не трогай по сети `cache.db` и базы `*.dat`/`*.mmdb` — mihomo держит их открытыми.
- После правки перезапусти прокси: `xkeen -restart`.

## Бэкап и восстановление

Флешки в роутере работают круглосуточно и со временем изнашиваются. Скрипт `backup.sh` снимает **умный образ** рабочей флешки (только занятые блоки ext4 — через `e2image`), чтобы при смерти накопителя восстановить рабочий Entware/XKeen на новую флешку за минуту, а не настраивать заново.

Образ компактный: с 32 ГБ флешки, где данных ~40 МБ, файл получится ~40 МБ (а не 32 ГБ), снятие — секунды.

Проще всего запустить `backup.sh` **без аргументов** — появится меню:
- **backup** — снять образ флешки в файл `.kbak`;
- **restore** — записать образ на флешку (скрипт сам покажет список найденных `.kbak`, имя вводить не нужно);
- **clone** — снять образ с одной флешки и сразу залить на другую (без промежуточного ввода имени файла).

```
bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/backup.sh)
```

Можно и по-старому, аргументом:

```
# снять образ:
bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/backup.sh) backup

# восстановить на флешку (будет стёрта); без имени файла — покажет меню выбора:
bash <(curl -fsSL https://raw.githubusercontent.com/lastik9/keenetic-entware/main/backup.sh) restore keenetic-backup-XXXX.kbak
```

Восстановление воссоздаёт разметку, разворачивает ext4 из образа, растягивает ФС на весь раздел и готовит swap-раздел. Клон переносит всё: Entware, пакеты, конфиги, init-скрипты, SSH-ключи.

> **После восстановления обязательно включи swap на роутере** — иначе XKeen упадёт. См. [«После restore XKeen падает с out of memory»](#после-restore-xkeen-падает-с-out-of-memory).

### Ужатие ФС при бэкапе

Перед снятием образа скрипт **временно ужимает** ext4 до реального объёма данных (`resize2fs`), снимает образ, и **сразу возвращает ФС на весь раздел**. Флешка-источник остаётся ровно такой, какой была.

Зачем: `dd` при восстановлении пишет образ до границы ФС. Если ФС занимает весь раздел, `dd` честно льёт десятки гигабайт, из которых почти всё — нули.

Реальные цифры с 64 ГБ флешки (данных ~1.1 ГБ):

| | без ужатия | с ужатием |
|---|---|---|
| ФС в образе | 56.3 ГиБ | **512 МБ** |
| сколько пишет `dd` | 56.3 ГиБ | **512 МБ** |
| файл `.kbak` | 75 МБ | 53 МБ |

Запись сокращается **в 113 раз**. Файл `.kbak` и так был компактным (`e2image` копирует только занятые блоки) — узким местом всегда было именно восстановление.

Ужатие включено по умолчанию и происходит, только если `e2fsck` подтвердил, что ФС чистая. Отключить:

```
NO_SHRINK=1 ./backup.sh backup
```

Если во время бэкапа что-то пойдёт не так (Ctrl-C, выдернули флешку), скрипт перехватит выход и растянет ФС обратно. Если даже это не удастся — данные целы, ФС просто меньше раздела; скрипт напечатает команду для ручного возврата.

### Клон на флешку меньшего объёма

С ужатием это стало возможно: важен не размер исходной флешки, а **влезут ли данные** в целевой раздел. Систему с 64 ГБ флешки, где занято 1.1 ГБ, можно развернуть на 16 ГБ — ФС потом растянется до нового размера.

Скрипт предупредит, что целевая флешка меньше, и проверит вместимость. Для **старых** образов (снятых до появления ужатия) правило прежнее: целевая флешка не меньше исходной.

### После restore XKeen падает с out of memory

Восстановление затирает swap-раздел, и метка `SWAP` вместе с сигнатурой исчезает. На свежей флешке свопа нет. Роутеру с 256 МБ RAM его не хватает ровно на разборе `category-ads-all` (164 тысячи правил), и `mihomo` падает:

```
INFO Load GeoSite rule: category-ads-all
fatal error: runtime: out of memory
```

Это **не повреждение образа**. Лечится одной командой на роутере — пункт **3) Только swap** в `router-setup.sh`:

```
wget -O /tmp/rs.sh https://raw.githubusercontent.com/lastik9/keenetic-entware/main/router-setup.sh && sh /tmp/rs.sh
```

Скрипт найдёт swap-раздел по разметке (метки-то нет), поставит метку и сигнатуру, включит своп и пропишет автозапуск. Проверить:

```
free | grep Swap
cat /proc/swaps
```

Дальше `xkeen -restart` — старт займёт полторы минуты, это нормально.

### Проверка ФС перед снятием образа

Перед `e2image` скрипт прогоняет `e2fsck -fn` (только чтение, ничего не меняет). Если флешку выдернули из роутера без размонтирования, журнал ext4 остаётся «грязным», и `e2image` может упасть:

```
e2image: Can't read next inode while getting next inode
```

Хуже, если образ снимется молча — в него уедет мусор. Поэтому при грязной ФС скрипт покажет вывод `e2fsck` и предложит: `[f]` починить (`e2fsck -fy`), `[c]` продолжить как есть, `[q]` выйти.

Выбрал `[c]` — ужатие ФС будет пропущено: `resize2fs` не работает с непроверенной ФС, и правильно делает.

**Про журнал ext4.** Если флешку вынули из роутера, который писал на неё до последней секунды, `e2fsck -fn` скажет:

```
Warning: skipping journal recovery because doing a read-only filesystem check.
```

В журнале остались незакоммиченные транзакции. Это норма, а не поломка. Починка (`-fy`) проиграет журнал, и **число файлов может слегка измениться** — например, применится отложенное удаление временных файлов. Так и должно быть.

### Где взять полный путь к `.kbak`

Если файл лежит не рядом со скриптом и не в `~/keenetic`, автопоиск его не найдёт — нужен **полный путь**, например:

```
/Users/<пользователь>/Desktop/keenetic-backup-20260709-0226.kbak
```

Набирать вручную не обязательно: **перетащи файл из Finder прямо в окно Терминала** — путь подставится сам. Либо перейди в нужную папку и посмотри `pwd`.

### Растянуть ФС на весь объём новой флешки

Делается **автоматически**: после `dd` скрипт прогоняет `e2fsck -fy` и `resize2fs`, ФС занимает весь целевой раздел. Ничего доделывать не нужно — скрипт так и напишет в конце.

Итоговый объём будет чуть меньше номинала: например, из 60.5 ГБ раздела получится ~55 ГБ. Это нормально — служебные структуры ext4 плюс 5% резерва под `root`.

Если `resize2fs` недоступен (старый бандл без него, и Homebrew не стоит), скрипт честно предупредит и оставит ФС исходного размера. Тогда растянуть вручную **на роутере**, обязательно **сначала `e2fsck -f`**:

```
opkg install e2fsprogs resize2fs
DEV=$(blkid | grep 'LABEL="OPKG"' | cut -d: -f1)
e2fsck -f "$DEV" && resize2fs "$DEV"
```

(раздел `OPKG` находится по метке — так же, как swap по метке `SWAP`.)

## Как это работает

- Разметка — встроенным `diskutil`; типы разделов (0x82 swap, 0x83 Linux) выставляются встроенным `fdisk`.
- ext4 создаётся через `mke2fs`, а установщик пишется в ext4-раздел через `debugfs` — **без монтирования** (macOS не умеет монтировать ext4, и это не требуется).
- Бэкап использует `e2image` (только занятые блоки). Восстановление разворачивает `e2image` в файл размером с ФС из образа и заливает его на раздел через `dd` **целиком, без** `conv=sparse` — пропуск нулевых блоков оставлял бы на их месте мусор от старой разметки и повреждал ФС. Ужатие ФС перед бэкапом безопасно и укорачивает `dd` по **границе ФС** — это принципиально не то же самое, что `sparse`.
- `resize2fs`, в отличие от `e2image`, работает с **блок**-устройством macOS (`/dev/diskNsX`) без нареканий.
- Размер ужатой ФС берётся **из вывода `resize2fs`**, а не из расчёта: он округляет запрошенный размер вниз до границы блок-группы (например, попросили 131550 блоков — получили 131072). Записать в метаданные расчётное число значило бы недописать хвост при `dd` и тихо испортить ФС.
- `resize2fs -P` (минимальный размер) систематически занижает, поэтому скрипт берёт запас: `+30%`, но не меньше `+8192` блоков.
- Ужатая ФС часто **меньше «занятого» объёма** — потому что вместе с пустыми блок-группами схлопываются их таблицы inode. На 64 ГБ флешке одни только таблицы занимали 942 МБ.
- `e2fsck` на macOS работает через **блок**-устройство (`/dev/diskNsX`), не через `rdisk`. Проверить готовый клон вручную:
  ```
  diskutil unmountDisk force /dev/diskN
  sudo e2fsck -fn /dev/diskNs2 | tail -15
  ```
  Здоровый вывод — 5 проходов без строк `Fix?` и `illegal block`, в конце `OPKG: NNNN/... files`.
- `e2image` **не умеет писать в raw-устройство macOS** (`/dev/rdiskNsX`) — выдаёт `block -1`. Поэтому restore сначала разворачивает образ в файл, и уже файл пишет на раздел через `dd`.
- Бинарники `mke2fs`/`debugfs`/`e2image` собраны из e2fsprogs как **universal** (arm64 + x86_64), внутренние либы влинкованы статически, поэтому зависят только от `/usr/lib/libSystem`. Как пересобрать самому — см. [BUILD.md](BUILD.md).
- `e2fsck` и `resize2fs` пока **не входят** в скачиваемый бандл: если установлен Homebrew с `e2fsprogs`, скрипт возьмёт их оттуда. Без Homebrew проверка ФС и ужатие будут пропущены — бэкап и восстановление отработают, просто медленнее и без предчека. Добавление обеих утилит в бандл — в планах.

## Собрать бинарники самому

См. [BUILD.md](BUILD.md). Коротко: `bash build-macos-e2fsprogs.sh` скачает исходники e2fsprogs, соберёт universal-бандл, проверит его и напечатает SHA-256.

## Безопасность

- В списке — **только съёмные** накопители, системный диск выбрать нельзя.
- Перед стиранием нужно вручную ввести точное имя устройства для подтверждения.
- Скачанный бандл бинарников проверяется по зафиксированному SHA-256.
- Смени пароль SSH по умолчанию (`passwd root`) — `router-setup.sh` предложит это сам.
