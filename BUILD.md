# BUILD.md — сборка universal-бинарников e2fsprogs для macOS

Этот документ описывает, как один раз собрать пять инструментов e2fsprogs
(universal: Apple Silicon + Intel) для пути «голого мака» — чтобы пользователю
**не требовался Homebrew**:

- `mke2fs` — создание ext4 (нужен `prepare.sh`);
- `debugfs` — запись установщика в ext4 без монтирования (`prepare.sh`);
- `e2image` — снятие «умного» образа: только занятые блоки (`backup.sh`);
- `e2fsck` — проверка ФС перед бэкапом и её починка (`backup.sh`);
- `resize2fs` — ужатие ФС перед снятием образа и растяжение при restore (`backup.sh`).

Всё автоматизировано скриптом `build-macos-e2fsprogs.sh`. Ниже — что он делает,
зачем, и как опубликовать результат.

## Зачем это нужно

macOS не умеет работать с ext4 своими средствами — нужны инструменты из e2fsprogs.
`prepare.sh` и `backup.sh` умеют брать их из Homebrew, но у «голого» пользователя
brew нет. Решение: положить готовые бинарники в GitHub Releases, а скрипты
скачивают их автоматически (через `curl`, поэтому карантин Gatekeeper не ставится)
и подписывают ad-hoc (`codesign -s -`, нужно на Apple Silicon).

Отдельно про `e2fsck` и `resize2fs`: без них `backup.sh` работает «по-старому» —
проверка ФС и ужатие молча пропускаются, поэтому раньше главная фича бэкапа
(маленький образ + быстрый restore, «Вариант 3») **не работала на маке без brew**.
Теперь все пять инструментов в бандле, и путь «голого мака» полноценный.

## Почему из исходников, а не из brew

- Бинарники из brew слинкованы **динамически** и тянут набор dylib'ов
  (`libext2fs`, `libcom_err`, `libe2p`, `libblkid`, `libuuid`). На маке без brew
  они падают с «library not loaded».
- macOS **не поддерживает** полностью статическую линковку (libSystem всегда
  динамический), поэтому «один статический файл» невозможен.
- Но: на macOS e2fsprogs по умолчанию **не собирает** shared-либы, значит
  внутренние либы влинковываются в бинарник статически, и снаружи остаётся
  зависимость только от `libSystem` (есть на любом маке). Это и есть цель.

## Что нужно на маке

- **Xcode Command Line Tools** (даёт `clang` и `make`). Проверить:
  ```
  xcode-select -p
  ```
  Если ругнётся — поставить: `xcode-select --install`.
  (Если у тебя уже есть Homebrew — CLT почти наверняка стоят.)
- Интернет (скрипт качает исходники e2fsprogs 1.47.4 с kernel.org).

## Как собрать

```
bash build-macos-e2fsprogs.sh
```

Скрипт по шагам:
1. Качает исходники `e2fsprogs-1.47.4` (kernel.org, фолбэк — SourceForge).
2. `configure` с `CC="clang -arch arm64 -arch x86_64"` — обе архитектуры сразу.
3. Собирает только нужное: библиотеки + `mke2fs` + `debugfs` + `e2image` +
   `e2fsck` + `resize2fs` (peripheral-инструменты вроде `e4defrag`/`fuse2fs`
   не трогаем — они на macOS могут не компилироваться и нам не нужны).
   `e2fsck` заметно крупнее прочих — его сборка идёт дольше, это норма.
4. Проверяет каждый бинарник: `file` (должно быть *2 architectures*) и
   `otool -L` (внешние зависимости — только `/usr/lib` и `/System`).
5. Пакует плоский `dist/e2fsprogs-macos-universal.tar.gz` (внутри — все пять
   инструментов в корне, как ждут `prepare.sh` и `backup.sh`).
6. Печатает `sha256` и готовые строки для вставки в `backup.sh` И `prepare.sh`.

Версия e2fsprogs (1.47.4) специально совпадает с той, что ставит Homebrew —
чтобы путь «через brew» и путь «голого мака» вели себя одинаково.

## Если что-то пошло не так

- Скрипт при падении печатает хвост соответствующего лога (`/tmp/e2fs-*.log`).
  Пришли его — разберёмся (как отлаживали `prepare.sh`).
- Если `otool -L` покажет постороннюю зависимость (например, из `/opt/homebrew`)
  — значит configure зачем-то подхватил shared-либы; напиши, поправим флаги.

## Публикация в GitHub Releases

1. Создай релиз с тегом, например `e2fsprogs-v1.47.4-macos`.
   При пересборке того же тега перезалей asset (в вебе: Edit release → удалить
   старый `.tar.gz` → приложить новый → Update; либо `gh release upload <тег>
   dist/e2fsprogs-macos-universal.tar.gz --clobber`).
2. Приложи файл `dist/e2fsprogs-macos-universal.tar.gz`.
3. Пропиши эти строки **в обоих файлах — `backup.sh` И `prepare.sh`**
   (скрипт печатает готовые значения):
   ```
   E2FS_OWNER_REPO="твой_гитхаб/keenetic-entware"
   E2FS_RELEASE_TAG="e2fsprogs-v1.47.4-macos"
   E2FS_BUNDLE_SHA256="<sha256 из вывода скрипта>"
   ```
   При простой пересборке под тем же тегом меняется только `E2FS_BUNDLE_SHA256`.
   Замену удобно делать по старому хешу, чтобы не промахнуться строкой:
   ```
   sed -i '' 's/<старый sha256>/<новый sha256>/' backup.sh prepare.sh
   ```
   (`sed -i ''` — синтаксис BSD sed на macOS, пустая строка после `-i`
   обязательна.) Проверка — новый хеш ровно по разу в каждом файле, старого нет:
   ```
   grep -c "<новый sha256>" backup.sh prepare.sh   # → по 1
   grep -c "<старый sha256>" backup.sh prepare.sh   # → по 0
   ```

## Как проверить путь «голого мака» после публикации

Не удаляя Homebrew, прогони оба скрипта с флагом, который его игнорирует.

`prepare.sh` (создание флешки) — должен скачать бандл, сверить sha256, подписать
бинарники и найти `mke2fs`/`debugfs`:
```
IGNORE_BREW=1 DRY_RUN=1 bash prepare.sh
```

`backup.sh` (бэкап + Вариант 3) — должен скачать тот же бандл и поднять из него
`e2image`, `e2fsck` и `resize2fs` (именно они раньше отсутствовали):
```
IGNORE_BREW=1 DRY_RUN=1 bash backup.sh backup
```
В выводе жди три зелёные строки с путями вида `.../T/tmp.*/bin/e2image`,
`.../bin/e2fsck`, `.../bin/resize2fs` — то есть инструменты берутся из бандла,
а не из `/opt/homebrew`. Дальше скрипт упрётся в поиск флешки — это ожидаемо.

Полную сквозную проверку (реальный бэкап с предчеком и ужатием без brew) можно
сделать на «расходной» флешке: `IGNORE_BREW=1 bash backup.sh backup`.

## Лицензия

e2fsprogs распространяется под GPL-2.0 (libss/libet — MIT-подобная).
При распространении бинарников приложи соответствующие тексты лицензий из
исходников (файлы `NOTICE`, `lib/ss/mit-sipb-copyright.h`, `lib/et/internal.h`).
