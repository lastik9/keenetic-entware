# BUILD.md — сборка universal-бинарников e2fsprogs для macOS

Этот документ описывает, как один раз собрать `mke2fs` и `debugfs`
(universal: Apple Silicon + Intel) для пути «голого мака» в `prepare.sh`
— чтобы пользователю **не требовался Homebrew**.

Всё автоматизировано скриптом `build-macos-e2fsprogs.sh`. Ниже — что он делает,
зачем, и как опубликовать результат.

## Зачем это нужно

macOS не умеет создавать ext4 своими средствами — нужен `mke2fs` из e2fsprogs.
`prepare.sh` умеет брать его из Homebrew, но у «голого» пользователя brew нет.
Решение: положить готовые бинарники в GitHub Releases, а `prepare.sh` скачивает
их автоматически (через `curl`, поэтому карантин Gatekeeper не ставится) и
подписывает ad-hoc (`codesign -s -`, нужно на Apple Silicon).

## Почему из исходников, а не из brew

- Бинарники из brew слинкованы **динамически** и тянут набор dylib'ов
  (`libext2fs`, `libcom_err`, `libe2p`, `libblkid`, `libuuid`). На маке без brew
  они падают с «library not loaded».
- macOS **не поддерживает** полностью статическую линковку (libSystem всегда
  динамический), поэтому «один статический файл» невозможен.
- Но: на macOS e2fsprogs по умолчанию **не собирает** shared-либы, значит
  внутренние либы влинковываются в бинарник статически, и снаружи остаётся
  зависимость только от `libSystem` (есть на любом маке). Это и есть цель.

## Что нужно на маке

- **Xcode Command Line Tools** (даёт `clang` и `make`). Проверить:
  ```
  xcode-select -p
  ```
  Если ругнётся — поставить: `xcode-select --install`.
  (Если у тебя уже есть Homebrew — CLT почти наверняка стоят.)
- Интернет (скрипт качает исходники e2fsprogs 1.47.4 с kernel.org).

## Как собрать

```
bash build-macos-e2fsprogs.sh
```

Скрипт по шагам:
1. Качает исходники `e2fsprogs-1.47.4` (kernel.org, фолбэк — SourceForge).
2. `configure` с `CC="clang -arch arm64 -arch x86_64"` — обе архитектуры сразу.
3. Собирает только нужное: библиотеки + `mke2fs` + `debugfs`
   (peripheral-инструменты вроде `e4defrag`/`fuse2fs` не трогаем — они на macOS
   могут не компилироваться и нам не нужны).
4. Проверяет каждый бинарник: `file` (должно быть *2 architectures*) и
   `otool -L` (внешние зависимости — только `/usr/lib` и `/System`).
5. Пакует плоский `dist/e2fsprogs-macos-universal.tar.gz` (внутри — `mke2fs`
   и `debugfs` в корне, как ждёт `prepare.sh`).
6. Печатает `sha256` и готовые строки для вставки в `prepare.sh`.

Версия e2fsprogs (1.47.4) специально совпадает с той, что ставит Homebrew —
чтобы путь «через brew» и путь «голого мака» вели себя одинаково.

## Если что-то пошло не так

- Скрипт при падении печатает хвост соответствующего лога (`/tmp/e2fs-*.log`).
  Пришли его — разберёмся (как отлаживали `prepare.sh`).
- Если `otool -L` покажет постороннюю зависимость (например, из `/opt/homebrew`)
  — значит configure зачем-то подхватил shared-либы; напиши, поправим флаги.

## Публикация в GitHub Releases

1. Создай релиз с тегом, например `e2fsprogs-v1.47.4-macos`.
2. Приложи файл `dist/e2fsprogs-macos-universal.tar.gz`.
3. В `prepare.sh` пропиши три строки (скрипт печатает готовые значения):
   ```
   E2FS_OWNER_REPO="твой_гитхаб/keenetic-entware-flash"
   E2FS_RELEASE_TAG="e2fsprogs-v1.47.4-macos"
   E2FS_BUNDLE_SHA256="<sha256 из вывода скрипта>"
   ```

## Как проверить путь «голого мака» после публикации

Не удаляя Homebrew, прогони `prepare.sh` с флагом, который его игнорирует:
```
IGNORE_BREW=1 DRY_RUN=1 bash prepare.sh
```
Скрипт должен скачать бандл из Releases, проверить sha256, подписать бинарники
и найти `mke2fs`/`debugfs` в них — то есть повести себя как на маке без brew.

исходников (файлы `NOTICE`, `lib/ss/mit-sipb-copyright.h`, `lib/et/internal.h`).
